# 言語ランタイムとビルドの仕組み — コンテナ開発の前提知識

Dev Containers やコンテナベースの開発環境を使うとき、「Node.js を入れる」「Java 17 を入れる」といった設定を書く。しかしそもそもこれは何をインストールしているのか。ランタイムとビルドツールの違いを理解すると、コンテナの構成が納得できるようになる。

## 言語ランタイムとは何か

プログラミング言語で書いたコードは、そのままではコンピュータは理解できない。コードを実行可能な形に変換し、実行する**エンジン**が必要になる。これが**言語ランタイム**（実行環境）だ。

```
あなたが書いたコード → [言語ランタイム] → コンピュータが実行
```

つまり「Node.js をインストールする」とは「JavaScript を動かすためのエンジンをインストールする」という意味であり、「Java 17 をインストールする」とは「Java を動かすためのエンジンのバージョン 17 をインストールする」という意味だ。

## 主要な言語ランタイムの仕組み

言語によってコードの実行方法が異なる。大きく3つのパターンがある。

### パターン1: インタプリタ型 — Python, Ruby, PHP

コードを1行ずつ読みながらその場で実行する。

```
ソースコード (.py) → [Python インタプリタ] → 実行
```

`python` コマンドがランタイムそのもの。コードを書いたら即実行できる手軽さがある。

### パターン2: 仮想マシン型 — Java (JVM), C# (.NET)

コードをまず**中間コード**に変換（コンパイル）し、それを**仮想マシン**が実行する。

```
ソースコード (.java)
    ↓ コンパイル（javac）
バイトコード (.class)
    ↓
  [JVM] → 実行
```

Java の場合、この仮想マシンが **JVM（Java Virtual Machine）** だ。「Java をインストールする」とは、実質的に「JVM をインストールする」ことを意味する。

なぜわざわざ中間コードを挟むのか？ JVM さえあれば、Windows でも Mac でも Linux でも同じバイトコードが動く。**「Write Once, Run Anywhere（一度書けばどこでも動く）」** というのが Java の設計思想だ。

JVM の上では Java 以外の言語も動く。Kotlin、Scala、Groovy なども JVM 向けにコンパイルされるため、「Java 17 をインストールする」と Kotlin も使える。

```
Java   ─┐
Kotlin ─┤→ バイトコード → [JVM] → 実行
Scala  ─┘
```

### パターン3: JavaScript ランタイム — Node.js, Deno, Bun

JavaScript はもともとブラウザの中で動く言語だった。Chrome などのブラウザには JavaScript エンジン（V8）が内蔵されており、Web ページ上の JavaScript はこれが実行している。

Node.js は、この **V8 エンジンをブラウザから取り出してサーバーでも使えるようにしたもの** だ。

```
ブラウザの場合:   JavaScript → [Chrome 内蔵の V8] → 実行
サーバーの場合:   JavaScript → [Node.js (V8)] → 実行
```

つまり「Node.js をインストールする」とは「ブラウザなしで JavaScript を実行できるエンジンをインストールする」ということ。これにより JavaScript で Web サーバーや CLI ツールが作れるようになった。

フロントエンド開発でも Node.js が必要なのは、ビルドツール（Vite, webpack 等）やパッケージマネージャ（npm）が Node.js 上で動くためだ。

## ランタイムのバージョン管理が重要な理由

ランタイムにはバージョンがあり、バージョンが違うと動作が変わる。

```
開発者A: Node.js 18 で開発 → 動く
開発者B: Node.js 22 で開発 → 新しい構文を使ってしまう
本番:    Node.js 18 で運用 → 開発者Bのコードが動かない
```

Dev Containers のようなコンテナ開発環境は、この問題を解決する。設定ファイルにランタイムとそのバージョンを明記することで、チーム全員が同じバージョンのランタイムで開発できる。

## ビルドツールとランタイムは別物

現場では「ビルド（コードを変換・コンパイルする工程）に必要なもの」と「実行に必要なもの」が異なることがよくある。

### Java: JDK と JRE

Java の世界では、この違いが明確に分かれている:

| | JDK（開発キット） | JRE（実行環境） |
|---|---|---|
| **含むもの** | コンパイラ（javac）+ JVM + 標準ライブラリ + デバッガ | JVM + 標準ライブラリ |
| **役割** | コードを書いて、コンパイルして、実行する | コンパイル済みのバイトコードを実行するだけ |
| **サイズ** | 大きい | 小さい |
| **使う場面** | 開発マシン | 本番サーバー |

```
開発時:  .java → [JDK の javac でコンパイル] → .class → [JDK の JVM で実行・デバッグ]
本番:                                          .class → [JRE の JVM で実行]
```

開発時は JDK が必要だが、本番には JRE だけあればいい。

### JDK と JRE のバージョンの関係

ここで注意すべきことがある。**JDK にはそのバージョンの JRE が含まれている**。JDK 21 をインストールすると JRE 21 がセットで入る。JDK 21 と JRE 17 を別々にインストールする、という構成にはならない。

```
JDK 21 をインストール
┌──────────────────────────┐
│ コンパイラ（javac）        │
│ デバッガ                   │
│ 開発ツール                 │
│ ┌──────────────────────┐ │
│ │ JRE 21（JVM + ライブラリ）│ ← JRE 21 がセットで入る
│ └──────────────────────┘ │
└──────────────────────────┘
  JRE 17 だけ別に入れる、という仕組みではない
```

では「本番は Java 17 で動いているが、開発では新しい JDK を使いたい」場合はどうするか。2つのアプローチがある。

**アプローチ1: 本番と同じバージョンの JDK を使う（安全・推奨）**

最もシンプルで確実。本番が Java 17 なら開発も JDK 17 を使う。本番と同じバージョンで開発すれば「開発では動くのに本番で動かない」が起きない。特に理由がなければこれが正解だ。

**アプローチ2: クロスコンパイル**

JDK 21 の新しい開発ツール（高速なコンパイル、改善されたデバッガ等）を使いたいが、本番は Java 17 という場合。JDK には**クロスコンパイル**の仕組みがある。JDK 21 でコンパイルしつつ「Java 17 でも動くバイトコードを出力しろ」と指示できる:

```xml
<!-- pom.xml (Maven の場合) -->
<properties>
    <maven.compiler.release>17</maven.compiler.release>
</properties>
```

```kotlin
// build.gradle.kts (Gradle の場合)
java {
    sourceCompatibility = JavaVersion.VERSION_17
    targetCompatibility = JavaVersion.VERSION_17
}
```

こうすると JDK 21 のコンパイラが「Java 17 の範囲の機能だけ使っているか」をチェックし、Java 17 互換のバイトコードを出力する。

```
開発環境 (JDK 21)
  .java → [JDK 21 の javac + --release 17] → Java 17 互換の .class

本番 (JRE 17)
  Java 17 互換の .class → [JRE 17] → 実行できる
```

Java 21 でしか使えない構文をうっかり書いてしまっても、コンパイル時にエラーになるため安全だ。

### ビルドツール自体の Java バージョン問題

しかしクロスコンパイルだけでは解決できないケースがある。Gradle や Maven **自身の実行**にも Java が必要で、そのバージョンがアプリと異なることがある:

```
Gradle 8.x:  実行に JRE 17 以上が必要
アプリコード:  Java 11 向けにコンパイル・実行したい
```

開発環境に JDK 17 を1つだけ入れた場合、こうなる:

```
Gradle の実行:    JDK 17 の JVM で動く     ← OK
コンパイル:       --release 11 で互換出力  ← OK
テスト実行:       JDK 17 の JVM で動く     ← 問題！
```

コンパイルはクロスコンパイルで対処できるが、**テスト（特に E2E テスト）はアプリを実際に起動する**。JDK 17 しかなければ、テストは JVM 17 上でアプリを実行することになる。本番は Java 11 なのに、テスト環境は Java 17 — これではバージョン差異に起因するバグを見逃す可能性がある。

### 解決策: Gradle Toolchains

Gradle にはこの問題を解決する **Toolchains** という仕組みがある。「Gradle 自身は JDK 17 で動くが、コンパイルとテスト実行には JDK 11 を使え」と指定できる:

```kotlin
// build.gradle.kts
java {
    toolchain {
        languageVersion.set(JavaLanguageVersion.of(11))  // コンパイル・テストは Java 11 で
    }
}
```

Gradle は指定されたバージョンの JDK を開発環境内から探し、なければ自動でダウンロードする。つまり開発環境内に JDK が2つ共存する:

```
開発環境 (JDK 17 ベース)
┌────────────────────────────────────────┐
│                                         │
│  JDK 17 ── Gradle 自身の実行に使う      │
│                                         │
│  JDK 11 ── Gradle が自動ダウンロード     │
│         ── コンパイルに使う              │
│         ── テスト実行に使う（JVM 11）    │
│                                         │
└────────────────────────────────────────┘
```

Gradle Toolchains が JDK 11 のダウンロードと切り替えを自動で行うため、開発環境の設定を複雑にする必要がない。E2E テストを実行すると、アプリは JVM 11 上で起動する。本番と同じバージョンでテストできる。

**Toolchains を使わず手動で管理する場合**

**SDKMAN!** で複数の JDK を手動インストールすることもできる:

```bash
# 複数バージョンをインストール
sdk install java 17.0.9-tem
sdk install java 11.0.21-tem

# バージョンを切り替え
sdk use java 11.0.21-tem    # Java 11 に切り替え
./gradlew test               # Java 11 でテスト実行

sdk use java 17.0.9-tem     # Java 17 に戻す
```

ただし手動切り替えは忘れやすい。Gradle Toolchains のほうがビルド設定に宣言的に書けるため確実だ。

### TypeScript → JavaScript

TypeScript はそのままでは実行できない。ビルド時に JavaScript に変換（トランスパイル）する:

```
開発時:  .ts → [tsc でトランスパイル] → .js → [Node.js で実行]
本番:                                   .js → [Node.js で実行]
```

ビルドには TypeScript コンパイラ（`tsc`）と Node.js の両方が必要だが、本番で動くのは変換後の JavaScript なので Node.js だけあればいい。`tsc` は本番サーバーには不要だ。

### Go / Rust: コンパイル言語

Go や Rust はコードを直接マシンが実行できるバイナリ（実行ファイル）にコンパイルする:

```
開発時:  .go → [Go コンパイラ] → バイナリ → 実行
本番:                           バイナリ → 実行（ランタイム不要）
```

ビルドには Go コンパイラが必要だが、できあがったバイナリは**ランタイムなしで単体で動く**。本番サーバーに Go をインストールする必要がない。

### フロントエンド（React, Vue 等）

フロントエンドのフレームワークも同様にビルドとランタイムが分かれる:

```
開発時:  .tsx + .scss → [Vite/webpack + Node.js でビルド] → .html + .js + .css
本番:                                                       .html + .js + .css を配信
```

ビルドには Node.js + ビルドツールが必要だが、本番で配信するのは静的ファイル（HTML/JS/CSS）だけだ。本番サーバー（Nginx 等）に Node.js は不要。

### まとめ

| 言語 | ビルドに必要なもの | 実行に必要なもの |
|---|---|---|
| Java / Kotlin | JDK（コンパイラ + JVM） | JRE（JVM のみ） |
| TypeScript | Node.js + tsc | Node.js |
| Go | Go コンパイラ | なし（バイナリ単体で動く） |
| Rust | Rust コンパイラ（rustc） | なし（バイナリ単体で動く） |
| React / Vue | Node.js + Vite 等 | なし（静的ファイルを配信するだけ） |
| Python | なし（インタプリタ型） | Python |

## 開発環境と本番環境の違い

### 開発環境にはビルドツールもランタイムも両方入れる

コンテナベースの開発環境（Dev Containers 等）では、コンテナの中で実際にコードを書いて、ビルドして、実行して、デバッグする。だから**ビルドツールもランタイムも両方必要**だ。

Java の場合で考えてみる。開発環境内での開発サイクルはこうなる:

```
コードを書く (.java)
    ↓
コンパイルする (javac)     ← ビルドツールが必要
    ↓
実行する (java -jar ...)   ← ランタイムが必要
    ↓
デバッガでブレークポイントを張る ← デバッガが必要
    ↓
テストを走らせる (mvn test)  ← テストフレームワークが必要
    ↓
コードを修正して繰り返す
```

すべての工程がコンテナの中で起きるので、開発コンテナには「開発に使うものを全部」入れる。Java なら JRE（実行だけ）ではなく JDK（コンパイラ + 実行環境 + デバッガ一式）を入れるのはこのためだ。

```
開発コンテナ（開発用）               本番コンテナ（実行用）
┌───────────────────────────────┐    ┌──────────────────┐
│ ビルドツール                    │    │                  │
│  コンパイラ（javac, tsc）       │    │ ランタイム (JRE)  │
│  ビルドシステム（Maven, Vite）  │    │ アプリのバイナリ   │
│ ランタイム (JVM)                │    │ 設定ファイル       │
│ デバッガ                        │    │                  │
│ リンター / フォーマッター        │    └──────────────────┘
│ テストフレームワーク             │      ← 最小限で軽い
│ ソースコード                    │
│ IDE 拡張機能                    │
└───────────────────────────────┘
  ← 全部入りで重い
```

本番コンテナとは設計思想が真逆になる:

| | 開発コンテナ | 本番コンテナ |
|---|---|---|
| **目的** | 開発しやすさ | セキュリティ・軽量さ |
| **中身** | ビルドツール + ランタイム + デバッガ + IDE 拡張 | ランタイム + アプリ本体のみ |
| **サイズ** | 大きくてよい | できるだけ小さく |
| **ベースイメージ** | Ubuntu / Debian ベース | Alpine / distroless / scratch |

開発コンテナが大きくなるのは問題ではない。開発者の手元（またはリモートの開発マシン）で動くもので、ユーザーに配布するわけではないからだ。

### 本番用コンテナとの使い分け

「開発はこのコンテナで、本番は別のコンテナで」という構成が一般的だ。本番用の Dockerfile はマルチステージビルドで「ビルド用ステージ」と「実行用ステージ」を分ける:

```dockerfile
# 本番用 Dockerfile（開発コンテナではなく CI/CD で使う）

# --- ビルドステージ: JDK でコンパイル ---
FROM eclipse-temurin:21-jdk AS build
WORKDIR /app
COPY . .
RUN ./gradlew build

# --- 実行ステージ: JRE だけでアプリを実行 ---
FROM eclipse-temurin:21-jre
COPY --from=build /app/build/libs/app.jar /app.jar
CMD ["java", "-jar", "/app.jar"]
```

プロジェクトのディレクトリ構成としては、開発環境の定義と本番の定義をそれぞれ独立して管理する:

```
your-project/
├── .devcontainer/
│   └── devcontainer.json      ← 開発環境の定義（JDK + デバッガ + 拡張機能）
├── Dockerfile                  ← 本番コンテナの定義（マルチステージビルド）
├── docker-compose.yml          ← 本番のサービス構成
└── src/
```

開発コンテナで開発 → コードを push → CI/CD が本番用 Dockerfile でビルド → 本番にデプロイ、という流れになる。開発コンテナのことは本番側からは一切見えない。
